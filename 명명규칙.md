# **표준 명명 규칙**

프로젝트의 **일관성**과 **가독성**을 높이기 위한 표준 명명 규칙입니다.
팀원 간의 효율적인 협업과 유지보수를 위해 본 규칙을 준수해야 합니다.

## **1. 일반 규칙**

- **카멜 케이스 (Camel Case)**: 변수, 메소드 이름에 사용합니다. (예:
  userName, getUserList)

- **파스칼 케이스 (Pascal Case)**: 클래스, 인터페이스 이름에 사용합니다.
  (예: UserController, UserService)

- **스네이크 케이스 (Snake Case)**: 상수, 데이터베이스 테이블/컬럼
  이름에 사용합니다. (예: MAX_SIZE, user_name)

- **의미 있는 이름**: 변수나 메소드의 목적을 명확히 나타내는 이름을
  사용합니다.

- **축약어 사용 지양**: 특별한 경우를 제외하고는 약어를 사용하지
  않습니다. (예: controller보다 ctrl 지양)

## **2. 패키지(Package) 명명 규칙**

- **기본 구조**: com.회사명.프로젝트명 또는 org.프로젝트명과 같은
  형식으로 시작합니다.

- **레이어별 분리**: 애플리케이션의 계층별로 패키지를 분리합니다.

  - com.example.project.controller: 웹 요청 처리

  - com.example.project.service: 비즈니스 로직

  - com.example.project.repository: 데이터 접근

  - com.example.project.domain: 도메인 모델 (엔티티)

  - com.example.project.dto: DTO (Request/Response)

  - com.example.project.config: 설정 관련 클래스

  - com.example.project.util: 유틸리티 클래스

  - com.example.project.client: FeignClient 인터페이스

## **3. 클래스(Class) 명명 규칙**

클래스 이름은 **파스칼 케이스**를 사용하며, 클래스의 역할을 명확히
나타내는 \*\*접미사(Suffixes)\*\*를 붙입니다.

- **컨트롤러**: XxxController (예: UserController, ProductController)

- **서비스**: XxxService (예: UserService, ProductService)

- **레포지토리/DAO**: XxxRepository, XxxDao (예: UserRepository,
  ProductDao)

- **도메인/엔티티**: Xxx (단수 명사) (예: User, Product, Order)

- **설정 클래스**: XxxConfig (예: SecurityConfig, DatabaseConfig)

- **유틸리티 클래스**: XxxUtil (예: DateUtil, StringUtil)

- **예외 클래스**: XxxException (예: ResourceNotFoundException,
  InvalidInputException)

## **4. DTO(Data Transfer Object) 명명 규칙**

DTO는 요청(Request)과 응답(Response)의 목적에 따라 명확하게 분리하여
사용합니다.

- **요청 DTO (Request DTO)**: 클라이언트로부터 데이터를 받을 때
  사용하며, **Req 접미사**를 붙입니다.

  - XxxCreateReq: 자원 생성 요청 시 (예: UserCreateReq)

  - XxxUpdateReq: 자원 수정 요청 시 (전체 또는 부분) (예: UserUpdateReq,
    ProductPartialUpdateReq)

  - XxxReq: 특정 동작을 위한 일반적인 요청 시 (예: LoginReq,
    PasswordChangeReq)

- **응답 DTO (Response DTO)**: 클라이언트에 데이터를 보낼 때 사용하며,
  **Res 접미사**를 붙입니다.

  - XxxRes: 단일 자원에 대한 응답 시 (예: UserRes, ProductRes)

  - XxxListRes: 여러 자원에 대한 응답 시 (예: UserListRes,
    ProductListRes)

    - 혹은 List\<XxxRes\> 형태로 직접 리스트를 반환할 수도 있습니다.

  - XxxDetailRes: 상세 정보 응답 시 (예: OrderDetailRes)

## **5. Rest Controller Method 명명 규칙**

RESTful API의 특징을 살려 **HTTP 메소드(GET, POST, PATCH, PUT, DELETE)의
의미**와 **자원(Resource)의 복수형 명사**를 활용하여 명확하고 예측
가능한 이름을 부여합니다. 메소드 이름은 해당 엔드포인트가 수행하는
동작을 직관적으로 나타내야 합니다.

### **5.1. 자원 기반 (Resource-Oriented) 명명**

컨트롤러 메소드 이름은 자원의 행위를 표현하는 데 집중합니다. URI와 HTTP
메소드 조합이 이미 행위를 나타내므로, 메소드 이름은 해당 자원에 대한
구체적인 작업을 명시하는 데 사용됩니다.

- **전체 자원 조회 (GET)**

  - **URI**: /resources

  - **좋은 예시**: getUsers(), getProducts()

    - 설명: **get 동사**와 **복수형 명사**를 사용하여 전체 자원을
      조회함을 명확히 합니다.

  - **나쁜 예시**: selectUserList(), showAllProducts()

    - 설명: 데이터베이스 쿼리성 이름(select)이나 UI 지향적인
      이름(show)은 RESTful 하지 않습니다.

- **단일 자원 조회 (GET)**

  - **URI**: /resources/{id}

  - **좋은 예시**: getUser(@PathVariable Long id),
    getProduct(@PathVariable Long id)

    - 설명: **get 동사**와 단수형 명사를 사용하여 특정 자원을 조회함을
      명시합니다. (ById는 URI에 ID가 이미 포함되어 있으므로 생략
      가능합니다.)

  - **나쁜 예시**: findUserOne(), getSingleProduct()

    - 설명: 모호하거나 불필요한 단어 사용은 지양합니다.

- **검색 및 정렬을 포함한 자원 조회 (GET)**

  - **URI**:
    /resources?keyword={keyword}&sort={field}&page={page}&size={size}

  - **좋은 예시**:

    - searchUsers(String keyword, String sort, Pageable pageable)

    - getProductsByCriteria(String name, String category, String sort,
      Pageable pageable)

    - 설명: **search 또는 getXxxByCriteria 동사**를 사용하여 검색/정렬
      기능을 명확히 합니다. 파라미터로 검색 키워드, 정렬 기준, 페이징
      정보 등을 받습니다.

  - **나쁜 예시**:

    - getFilteredUsers()

    - findProductsWithSorting()

    - 설명: 메소드 이름만으로 어떤 검색/정렬 기준이 적용되는지
      불명확합니다.

- **자원 생성 (POST)**

  - **URI**: /resources

  - **좋은 예시**: createUser(@RequestBody UserCreateReq request),
    createProduct(@RequestBody ProductCreateReq request)

    - 설명: **create 동사**를 사용하여 자원 생성을 명확히 합니다. 요청
      DTO의 이름도 목적을 분명히 나타내는 것이 좋습니다.

  - **나쁜 예시**: saveUser(), postProduct()

    - 설명: save는 생성/수정 모두에 사용될 수 있어 모호하며, post는 HTTP
      메소드를 그대로 사용하므로 불필요합니다.

- **자원 전체 수정 (PUT)**

  - **URI**: /resources/{id}

  - **좋은 예시**: updateUser(@PathVariable Long id, \@RequestBody
    UserUpdateReq request), replaceProduct(@PathVariable Long id,
    \@RequestBody ProductReplaceReq request)

    - 설명: **update** 또는 replace 동사를 사용하여 자원 전체를
      수정(교체)하는 행위를 나타냅니다.

  - **나쁜 예시**: putUser(), changeProduct()

    - 설명: put은 HTTP 메소드를 그대로 사용하며, change는 너무
      일반적입니다.

- **자원 부분 수정 (PATCH)**

  - **URI**: /resources/{id}

  - **좋은 예시**: patchUser(@PathVariable Long id, \@RequestBody
    UserPartialUpdateReq request), modifyProduct(@PathVariable Long id,
    \@RequestBody ProductPartialUpdateReq request)

    - 설명: **patch 동사** 또는 \*\*modify\*\*와 함께 **부분적
      수정**임을 나타내는 이름을 사용합니다.

  - **나쁜 예시**: updatePartialUser(), editProductPatch()

    - 설명: 불필요하게 길거나 patch를 중복해서 사용하지 않습니다.

- **자원 삭제 (DELETE)**

  - **URI**: /resources/{id}

  - **좋은 예시**: deleteUser(@PathVariable Long id),
    removeProduct(@PathVariable Long id)

    - 설명: **delete 동사** 또는 remove를 사용하여 자원 삭제를 명확히
      합니다.

  - **나쁜 예시**: deleteUserById(), removeProductItem()

    - 설명: 불필요한 반복이나 과도한 정보를 포함하지 않습니다.

### **5.2. 특별한 경우 (Custom Actions)**

자원-중심적인 RESTful 원칙을 따르기 어려운 특정 비즈니스 로직의 경우,
동사를 사용하여 행위를 명시합니다. 이때도 가급적 HTTP 메소드와 충돌하지
않도록 명사 형태로 변환하거나 명확한 동사를 사용합니다.

- **좋은 예시**:

  - changeUserPassword(@PathVariable Long id, \@RequestBody
    PasswordChangeReq request) (POST /users/{id}/password)

  - activateAccount(@PathVariable Long id) (POST
    /accounts/{id}/activate)

    - 설명: 특정 비즈니스 행위를 나타내기 위해 동사를 명시적으로
      사용합니다. URI도 해당 행위를 반영하도록 구성하는 것이 좋습니다.

- **나쁜 예시**:

  - doSomethingAboutUser()

  - userProcess()

    - 설명: 추상적이거나 너무 일반적인 이름은 피해야 합니다.

## **6. FeignClient 및 DTO 명명 규칙**

외부 서비스와의 통신을 위한 FeignClient와 관련된 클래스 및 DTO 명명
규칙입니다.

### **6.1. FeignClient 인터페이스**

- **명명 규칙**: XxxClient 또는 XxxServiceClient

  - **좋은 예시**: UserServiceClient, ProductApiFeignClient

    - 설명: 어떤 외부 서비스를 호출하는지 명확하게 나타냅니다.
      FeignClient 접미사를 붙여 Feign 클라이언트임을 명시할 수 있습니다.

  - **나쁜 예시**: UserFeign, ProductCaller

    - 설명: 명확성이 떨어지거나 역할이 불분명합니다.

### **6.2. FeignClient DTO**

FeignClient를 통해 통신하는 DTO는 내부 서비스의 DTO와는 별도로 관리하는
것이 좋습니다. 이는 외부 API의 변경이 내부 도메인 모델에 직접적인 영향을
주지 않도록 하기 위함입니다. Req 및 Res 접미사를 사용하여 구분합니다.

- **패키지**: com.example.project.client.dto 또는
  com.example.project.client.서비스명.dto

- **명명 규칙**: 호출하는 외부 API의 맥락을 반영하여 DTO의 목적을 명확히
  합니다.

  - **좋은 예시**:

    - ExternalUserCreateReq: 외부 사용자 생성 요청 DTO

    - ExternalUserRes: 외부 사용자 응답 DTO

    - AuthServiceLoginReq: 인증 서비스 로그인 요청 DTO

    - AuthServiceTokenRes: 인증 서비스 토큰 응답 DTO

  - **나쁜 예시**:

    - UserDto (내부 DTO와 혼동 가능성)

    - DataResponse (너무 일반적)

## **7. 변수(Variable) 명명 규칙**

- **지역 변수**: **카멜 케이스**를 사용하며, 짧고 의미를 알 수 있는
  이름을 사용합니다. (예: count, result, temp)

- **멤버 변수**: **카멜 케이스**를 사용하며, 클래스 내에서 변수의 역할을
  명확히 합니다. (예: private String userName;)

- **상수(Constants)**: **대문자 스네이크 케이스**를 사용하며, static
  final로 선언합니다. (예: public static final int MAX_ATTEMPTS = 5;)

- **컬렉션 변수**: 복수형 명사를 사용하여 컬렉션임을 나타냅니다. (예:
  List\<User\> users, Map\<String, Product\> productsMap)

## **8. 파일 및 디렉토리 명명 규칙**

- **Java 소스 파일**: 클래스 이름과 동일하게 **파스칼 케이스**를
  사용합니다. (예: UserController.java)

- **설정 파일**: application.yml, application-dev.yml 등 일관된 패턴을
  유지합니다.

- **템플릿 파일**: index.html, login.html 등 소문자 스네이크 케이스 또는
  하이픈 케이스를 사용합니다.

## **9. 데이터베이스(Database) 관련 명명 규칙 (선택 사항)**

JPA/Hibernate를 사용하는 경우 엔티티 클래스 이름이 테이블/컬럼 이름으로
매핑되므로, 이 규칙은 선택 사항이거나 ORM 설정에 따라 달라질 수
있습니다.

- **테이블명**: snake_case, 복수형 명사를 사용하는 것이 일반적입니다.
  (예: users, products)

- **컬럼명**: snake_case, 단수형 명사를 사용합니다. (예: user_id,
  product_name, created_at)

- **관계 컬럼**: 참조_테이블명_id (예: user_id on orders table)

- **인덱스명**: idx_테이블명_컬럼명 (예: idx_users_email)

**10. 기타 명명 규칙**

- **어노테이션 (Annotation)**: 명사 또는 형용사 (예: \@Valid,
  \@Component, \@Configuration)

- **Enum**: **파스칼 케이스**를 사용하며, 각 Enum 값은 **대문자 스네이크
  케이스**를 사용합니다. (예: OrderStatus { PENDING, SHIPPED, DELIVERED
  })

**11. 명명 규칙 적용 및 강제 방안**

- **코드 리뷰**: 개발자가 작성한 코드가 명명 규칙을 준수하는지 코드
  리뷰를 통해 확인합니다.

- **정적 분석 도구**: Checkstyle, SonarQube 등 정적 분석 도구를 활용하여
  자동으로 명명 규칙 위반을 감지하고 리포트합니다.

- **개발 가이드**: 새로운 팀원 합류 시 본 명명 규칙 문서를 제공하고
  교육하여 규칙 준수를 독려합니다.
